# ROUTING-A Result

## Root cause vs. lead's brief

The lead's brief assumed the "[INFO] Routing task" + "Primary Recommendation"
output came from `~/.claude/hooks/agent-router.sh`. Investigation showed the
ACTUAL matcher producing that output is `~/.claude/helpers/router.js`,
generated by `generateAgentRouter()` in
`v3/@claude-flow/cli/src/init/helpers-generator.ts:254`. The
`UserPromptSubmit` hook in `~/.claude/settings.json:325-348` calls
`hook-handler.cjs route` which loads `router.js` — agent-router.sh is NOT
wired into settings (it would need to be added separately).

The "always coder" failure was inside `router.js`: the iterating-regex loop
fired the FIRST pattern that matched, and the first pattern was
`'implement|create|build|add|write code': 'coder'`. Any prompt with
"implement" routed to coder regardless of language/domain signals.

ROUTING-A fixes both layers:
- the live JS router (cause of the production failure),
- the bash hook script (defensive — for when it gets wired in too),
- the SwarmOps generator (so all users get the fix on their next `ruflo init`).

## Files modified

- **`~/.claude/helpers/router.js`** — live JS router. Replaced the first-match
  loop with a layered scorer (priority 100→20). Specialist boost suppresses
  generics when ANY non-generic agent matches. Always emits JSON, even for
  empty input. **35 patterns** (up from 8), **~6 generic verbs** demoted to
  last-resort priority 20.

- **`~/.claude/hooks/agent-router.sh`** — local bash hook. Mirrors the same
  layered model with separate `SPECIALIST_MATCHES`/`GENERIC_MATCHES` arrays;
  generics are suppressed entirely when any specialist matched. **34 match
  calls** (up from 21), with all language/framework/domain layers added.

- **`v3/@claude-flow/cli/src/init/helpers-generator.ts`** —
  `generateAgentRouter()` rewritten with the same scorer so `ruflo init`
  ships the improved `router.js` to all users. Backslash escaping audited:
  4 backslashes in TS source → 2 in emitted JS string → 1 in compiled regex.

- **`v3/@claude-flow/cli/__tests__/agent-router-patterns.test.ts`** — new
  vitest suite with **28 tests** covering language specialists, frameworks,
  domain-of-work, project domains, fallbacks, specialist boost, confidence
  calibration, and the deployed bash hook (gated on file existence).

## Specialist-boost logic

```javascript
const GENERIC_AGENTS = new Set(['coder', 'tester', 'reviewer', 'general-purpose']);

const hasSpecialist = hits.some((h) => !GENERIC_AGENTS.has(h.agent));
const eligible = hasSpecialist
  ? hits.filter((h) => !GENERIC_AGENTS.has(h.agent))
  : hits;

eligible.sort((a, b) => b.priority - a.priority);
const winner = eligible[0];
```

## Performance

- **bash hook**: ~70ms wall-clock (well under 100ms budget; ~30 subshell forks for `match` calls). Previous baseline (existing 21-pattern script) measured at the same ~70ms — adding 13 patterns did NOT increase latency because each pattern is one cheap `grep -qE`.
- **JS router**: ~40ms wall-clock (Node startup dominates). Pre-compiled regexes via `COMPILED_PATTERNS = PATTERNS.map(...)` mean per-call matching is sub-millisecond. Previous baseline ~35ms for 8 patterns.

Both well within the `timeout: 10000` configured for the `route` hook in
`settings-generator.ts:444`.

## Notes

- **Backslash escaping in `helpers-generator.ts`** is the trickiest part of
  this change. The function returns a TS template literal containing a JS
  source string containing JS string literals containing regex source. To
  emit a single `\` in the regex (for `\b`), the TS source needs FOUR
  backslashes: `\\\\b`. I confirmed parity by generating the file via tsx
  and running the same 13-prompt suite against the emitted JS.
- **No `system-architect` agent file is installed locally** — only
  `arch-system-design`. The router still recommends `system-architect`
  because that's the canonical name in the lead's CLAUDE.md vocabulary; the
  recommendation surfaces the right specialist, the lead spawns whatever
  installed agent matches.
- **`hooks_route` MCP tool at `mcp-tools/hooks-tools.ts:908+`** was NOT
  touched (coder-routing-helper's territory). The two layers are
  independent: this PR fixes the prompt-submit-time recommendation;
  hooks_route handles in-tool routing for `Agent.spawn` calls.
- **`agent-router.sh` is NOT currently emitted by SwarmOps init** — it's a
  user-local script the lead added manually. I did not add an emitter for it
  because (a) the JS router serves the same purpose at lower latency, and
  (b) emitting two routers via two different code paths would create a
  consistency burden. The bash script remains as a defensive backup if/when
  the lead wires it into settings.json.
- **`fix STRAT-1 install context` and `Gap 1 trace renderer` route to
  general-purpose** because the prompts contain ZERO language/framework/
  domain tokens — the lead's expectation that these route to
  `system-architect`/`typescript-expert` requires session context the matcher
  doesn't have. If desired, a follow-up could add CLAUDE.md project-specific
  patterns (e.g. `STRAT-\d+` → system-architect for SwarmOps).
- **Action-verb fallback NOT removed** — kept as the priority-20 last resort
  so prompts like `implement create build add` (pure verbs, zero specialist
  signal) still get a sensible routing instead of silent no-op.
